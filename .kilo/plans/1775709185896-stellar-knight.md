# Bot 项目 OpenAI 兼容端口排查与修复计划

## 范围修正

- 本计划仅针对 bot 主链路，也就是 `src/chat/**` 下机器人聊天与 AI Provider 集成部分。
- 不涉及仓库内其他子系统，也不把问题归因到非 bot 模块。

## 现状判断

- Bot 项目内的 OpenAI 兼容调用核心位于 `src/chat/services/ai/providers/openai_provider.py`，整体已经支持 `/chat/completions`、`/embeddings`、OpenAI 风格消息、工具调用和多模态 `image_url` 输入。
- 从 bot 主链路来看，更明显的问题不一定是 Provider 请求体本身，而是“模型选择 -> Provider 解析 -> Prompt/工具格式 -> 实际调用”这条链路没有完全打通，导致 OpenAI 兼容端点可能没有稳定命中。

## 已确认的关键问题

1. **聊天设置保存时丢失 provider 信息**
   - `AIModelSettingsView` 明明产出的是 `provider:model` 格式：`src/chat/features/chat_settings/ui/ai_model_settings_view.py:240`
   - 但真正保存时只保存了纯模型名：`src/chat/features/chat_settings/ui/chat_settings_view.py:376`
   - 结果：后续读取当前模型时，经常只拿到 `gpt-4o` 这种裸模型名，没有 `openai_compatible:gpt-4o` 这样的显式 provider。

2. **`chat_service` 用原始 model id 直接查 `_model_to_provider`，对新格式不兼容**
   - 当前代码：`src/chat/services/chat_service.py:289`
   - `_model_to_provider` 的 key 是纯模型名，不是 `provider:model`。
   - 如果数据库里将来保存 `openai_compatible:gpt-4o`，这里会查不到 provider，继而影响：
     - Prompt 输出格式判断：`src/chat/services/chat_service.py:298`
     - 工具格式判断：`src/chat/services/chat_service.py:332`

3. **UI 与 Service 已经部分支持 `provider:model`，但调用端没统一使用**
   - 解析完整模型 ID：`src/chat/features/chat_settings/ui/ai_model_settings_view.py:252`
   - Service 也支持读取/保存完整模型 ID：`src/chat/features/chat_settings/services/chat_settings_service.py:324`、`src/chat/features/chat_settings/services/chat_settings_service.py:364`
   - 但设置页面仍调用旧接口 `set_ai_model(model)`：`src/chat/features/chat_settings/ui/chat_settings_view.py:377`
   - 这说明兼容链路做到一半，最后落库环节没接上。

4. **bot 项目的示例配置没暴露 OpenAI 兼容端点配置，容易造成“看起来支持、实际没配”**
   - 当前仓库里的 bot 配置链路已读取 `OPENAI_COMPATIBLE_API_KEY` / `OPENAI_COMPATIBLE_URL`：`src/chat/services/ai/config/providers.py:182`
   - 但 bot 项目的示例配置 `D:\git\Odysseia-Guidance\.env.example` 中没有这两项，配置可发现性差。

## 代码层面的兼容性评估

### Bot 主链路里已经适配得比较完整的部分

- Provider 初始化：`src/chat/services/ai/service.py:127`
- Provider 配置读取：`src/chat/services/ai/config/providers.py:181`
- OpenAI 消息转换：`src/chat/services/ai/providers/openai_provider.py:331`
- Prompt 输出为 OpenAI 格式：`src/chat/services/prompt_service.py:1187`
- 工具转换为 OpenAI 格式：`src/chat/features/tools/services/tool_service.py:190`
- OpenAI 工具调用结果解析：`src/chat/services/ai/utils/tool_converter.py:323`
- 多模态 `image_url` 兼容：`src/chat/services/prompt_service.py:1248`、`src/chat/services/ai/service.py:822`

### Bot 主链路里仍存在的潜在兼容坑

- `openai_provider.py` 默认 `base_url` 带 `/v1`，请求路径是 `/chat/completions` 与 `/embeddings`。这要求外部端点必须和 OpenAI 官方一样把 `/v1` 放在 base URL 里；如果第三方端点只给根 URL，则会 404。当前没有做 base URL 归一化。
- `models_config.json` 没有任何 provider 为 `openai_compatible` 的模型定义，因此 OpenAI 兼容模型更多依赖 `providers.py` 里的硬编码模型列表；这会让设置页分组、描述、能力标记不完整。
- `openai_provider.py` 会直接透传 `response_format`，但没有针对不同第三方兼容端点做能力降级；某些兼容服务只接受 `{"type":"json_object"}` 或根本不支持该字段。
- `_convert_messages_to_openai_format()` 会把多段 `content` 中的图片抽掉，仅保留文字：`src/chat/services/ai/providers/openai_provider.py:429`。当前主链路通常会在 `prompt_service` 先转成标准 OpenAI 多模态格式，但如果别处直接传入 OpenAI 多段消息给 provider，这里会丢图。

## 建议实施步骤

1. **先把聊天面板做成真正可切换、可验证的模型控制台**
   - 目标不是只修一个 OpenAI 兼容 provider，而是让 bot 聊天面板能展示“当前所有可调用模型”，并允许自由切换。
   - 当前面板入口在 `src/chat/features/chat_settings/ui/chat_settings_view.py:160`，模型选择视图在 `src/chat/features/chat_settings/ui/ai_model_settings_view.py:17`。
   - 现状问题：模型列表主要来自 `models_config.json` 分组，不能可靠反映“当前接口真的可调用哪些模型”；这正是你说的“不知道有没有调用成功”的根源。

2. **新增“动态拉取可调用模型”能力**
   - 在 bot 项目里增加一个统一的模型发现层，按 provider 拉取真实可用模型，而不是只依赖静态配置。
   - 推荐设计：
     - `gemini_official` / 自定义 Gemini：如果 SDK/端点支持列模型则调用远端，否则回退本地配置。
     - `deepseek` / `openai_compatible`：优先调用兼容端点的 `GET /models`。
     - 若远端拉取失败，则显示“静态配置模型 + 状态未知”，避免面板空白。
   - 返回的数据结构建议统一为：`provider_name`, `model_id`, `display_name`, `supports_tools`, `supports_vision`, `status`, `source`。

3. **区分“已配置模型”和“远端真实模型”**
   - 当前 `chat_settings_service.get_available_models_with_provider()` 主要从 `models_config.json` 和数据库记录合并：`src/chat/features/chat_settings/services/chat_settings_service.py:526`
   - 这适合做“历史/配置展示”，但不适合做“可切换模型源”。
   - 建议改为双轨：
     - 静态配置模型：用于描述、参数默认值、人设配置。
     - 动态发现模型：用于聊天面板切换候选。
   - 当远端发现到新模型但本地没有配置时，也应该允许切换，只是采用默认能力标记和默认生成参数。

4. **修正模型设置保存逻辑**
   - 将 `src/chat/features/chat_settings/ui/chat_settings_view.py:377` 改为调用 `set_ai_model_with_provider(provider, model)`。
   - 目标是确保数据库稳定保存 `provider:model`。

5. **统一 `chat_service` 的 provider 解析逻辑**
   - 在 `src/chat/services/chat_service.py` 中，不再用 `ai_service._model_to_provider.get(current_model)` 直接查。
   - 改为先走 `ai_service.parse_model_id(current_model)`，若有显式 provider 则优先使用，否则再回退到 `_model_to_provider[model_name]`。
   - 这样 `output_format` 与 `get_dynamic_tools_for_context()` 都能拿到正确 provider。

6. **给聊天面板增加“当前模型状态”和“拉取结果来源”展示**
   - 面板上至少应显示：
     - 当前生效模型：`provider:model`
     - 模型来源：`remote` / `static_fallback`
     - 最近拉取状态：成功 / 失败
     - 失败原因摘要（如 `/models` 401、404、超时）
   - 这样用户能直接判断：
     - 面板里看到的是不是接口真实返回的模型
     - 当前切换到底有没有依据

7. **补齐 OpenAI 兼容模型定义，但降级为“增强项”**
   - 在 `src/chat/services/ai/config/models_config.json` 中补入 `openai_compatible` provider 的模型项，例如 `gpt-4o` / `gpt-4`。
   - 补上 `display_name`、`description`、`supports_vision`、`supports_tools` 等元信息，避免设置面板和能力判断不一致。
   - 但在新的设计下，静态配置不再是唯一数据源，只是元数据增强层。

8. **增强 OpenAI 兼容 base URL 容错**
   - 在 `src/chat/services/ai/providers/openai_provider.py` 增加 base URL 规范化逻辑。
   - 推荐策略：支持用户填写根 URL 或 `/v1` URL，内部统一拼成正确 endpoint，避免因 `/v1` 重复或缺失导致 404。

9. **为 OpenAI 兼容 provider 增加“模型探测”接口**
   - 除了 `generate()` / `generate_embedding()`，还需要新增类似 `list_models()` 的只读能力。
   - 典型实现：请求 `GET /models`，解析返回的 `data[].id`。
   - 如果第三方兼容端点没有实现 `/models`，则明确记录失败原因，并回退静态模型列表。

10. **增强对第三方兼容端点的请求兼容性**
   - 针对 `response_format`、图像输入、多模态 content 做更稳健判断。
   - 至少保证：
     - 若 `content` 为 OpenAI 多段格式，保留 `image_url` 而不是只提取文本。
     - 可选地对不支持的 `response_format` 做开关或降级。

11. **补充配置文档**
   - 在 `.env.example` 中加入：
     - `OPENAI_COMPATIBLE_API_KEY`
     - `OPENAI_COMPATIBLE_URL`
   - 让 OpenAI 兼容端口的配置方式和 Gemini / DeepSeek 一样可见。

12. **增加最小可观察性，确认“到底有没有调成功”**
   - 在 bot 主链路日志里增加一次清晰日志：
     - 当前选择的 `provider:model`
     - 本次实际命中的 provider
     - 实际请求使用的 base_url
     - 若支持的话，远端返回的 model id
   - 这样即使不看数据库，也能从日志判断切换是否真正生效。

## 建议验证项

- 聊天面板能显示每个 provider 的“远端拉取模型列表”与来源状态。
- 在设置页选择 OpenAI 兼容模型后，数据库中保存的是 `openai_compatible:gpt-4o` 而不是 `gpt-4o`。
- `chat_service` 日志中能正确打印解析后的 provider，且 `output_format=openai`。
- 切换模型后，日志里能看到本次请求实际命中的 `provider:model`，可以确认不是 UI 假切换。
- 发送普通文本、工具调用、带图片输入三类消息时，请求都能正确发往 OpenAI 兼容端点。
- 若使用第三方兼容服务，确认不会因 base URL 的 `/v1` 差异出现 404。

## 推荐优先级

- P0：面板支持动态拉取真实可调用模型 + 保存完整模型 ID + 修正 `chat_service` provider 解析
- P1：OpenAI 兼容 provider 增加 `/models` 探测 + 面板显示来源/状态
- P1：补 `.env.example`
- P2：补齐 `models_config.json` 中的 OpenAI 兼容模型定义
- P2：增强 `openai_provider.py` 的 base URL 和多模态容错

## 对 bot 项目根因的结论

- 当前 bot 项目里"OpenAI 兼容端口没适配好"的核心问题，可以更直接地表述为：**聊天面板没有把"真实可调用模型发现"和"实际生效切换"做好**。
- 也就是说，问题不只是 OpenAI Provider 请求格式，而是 UI、模型来源、provider 解析、日志可观察性四者没有闭环。
- 如果只修 Provider 本身而不修模型 ID 保存与解析，问题大概率仍会反复出现。

## 新增需求：Discord UI 面板直接填写 API 凭据

### 需求描述

在 Discord 聊天设置面板中，增加一个"🔌 配置 API 端点"按钮，允许有权限的管理员直接在 Discord 的 Modal 表单里填写 OpenAI 兼容端点的 URL、API Key 和可选备注名称，保存后立即生效，无需重启 bot、无需改 `.env`。

### 安全性分析

| 风险点 | 评估 | 结论 |
|--------|------|------|
| Discord Modal 传输 | ephemeral（只有触发者可见）+ HTTPS 加密 | ✅ 安全 |
| 存储（SQLite `global_settings`） | 明文存储，与 `.env` 文件风险等级相同 | ✅ 可接受（自托管场景） |
| 日志泄露 | 现有 `set_global_setting` 会将 value 打进日志 | ⚠️ **必须修复**：对 API Key 类 key 做脱敏 |
| 访问控制 | 设置面板已有 `is_authorized` 把关 | ✅ 已有保护 |
| Discord 消息历史 | 所有面板消息均为 ephemeral，不留历史 | ✅ 安全 |

**总结**：在自托管场景下没有新增风险，但日志脱敏是必须做的一处修复。

### 设计方案

#### 一、数据存储

复用现有的 `global_settings` SQLite 表（key/value 格式），新增以下 key：

| key | 说明 | 示例 |
|-----|------|------|
| `openai_compatible_url` | API 基础 URL | `https://openrouter.ai/api/v1` |
| `openai_compatible_api_key` | API Key（明文存储） | `sk-xxxx` |
| `openai_compatible_name` | 可选显示名称 | `OpenRouter` |

配置优先级（高→低）：**数据库 > 环境变量**

即：若数据库有值则优先使用；若没有则回退到 `.env`；两者都没有则该 provider 不加载。

#### 二、Provider 热重载

保存凭据后，需要立即用新配置替换 `ai_service` 中已运行的 provider 实例，而不是等重启。

在 `AIService` 中新增方法：

```python
def reload_openai_compatible_provider(self, api_key: str, base_url: str, name: str = "openai_compatible"):
    """用新凭据重建 OpenAI 兼容 Provider 并注册到 _providers，同时更新 _model_to_provider 映射。"""
```

调用链：UI 提交 → `ChatSettingsService.save_openai_compatible_config()` → `ai_service.reload_openai_compatible_provider()` → 模型发现刷新。

#### 三、UI 布局

在 `ChatSettingsView` 的第 4 行（Embedding 同行）增加一个按钮：

```
[🧠 Embedding模型]  [🔧 全局工具设置]  [🔌 API 端点配置]
```

点击"🔌 API 端点配置"后弹出 Modal（`discord.ui.Modal`），包含：

| 字段 | 类型 | 说明 |
|------|------|------|
| API 基础 URL | `TextInput` (short) | 预填当前值，placeholder 示例 URL |
| API Key | `TextInput` (short) | 预填 `已配置（留空不更改）` 或空，实际内容不回显 |
| 显示名称（可选） | `TextInput` (short) | 用于面板展示，默认 `OpenAI 兼容` |

Modal 提交后：
1. 验证 URL 格式（非空、以 http/https 开头）
2. 若 API Key 字段非空则更新；若为空则保留当前值
3. 写入数据库
4. 调用 `ai_service.reload_openai_compatible_provider()`
5. 返回确认 embed，显示：新 URL、Key 已更新/保留、当前连接状态（可选触发一次 `/models` 探测）

#### 四、需要修改的文件

| 文件 | 改动内容 |
|------|---------|
| `src/chat/utils/database.py` | `set_global_setting()` 对敏感 key 做日志脱敏（`api_key` 类的 value 只打后 4 位） |
| `src/chat/services/ai/config/providers.py` | `get_provider_configs()` 改为先查数据库再回退 env var（需要异步适配，或提供同步预加载路径） |
| `src/chat/services/ai/service.py` | 新增 `reload_openai_compatible_provider(api_key, base_url, name)` 方法 |
| `src/chat/features/chat_settings/services/chat_settings_service.py` | 新增 `save_openai_compatible_config()` 和 `get_openai_compatible_config()` |
| `src/chat/features/chat_settings/ui/chat_settings_view.py` | 新增按钮 + `on_api_endpoint_settings()` 回调 |
| `src/chat/features/chat_settings/ui/api_endpoint_settings_modal.py` | 新建 Modal 类 |

#### 五、providers.py 的异步问题处理

**已确认选用方案 A（最简单）**：

- 保持同步初始化不变，**不改 bot 启动流程**
- `save_openai_compatible_config()` 只做两件事：写 DB + 调用 `reload_openai_compatible_provider()`（不经过 providers.py，直接传参重建 Provider 实例）
- DB 中的配置作为"运行时热重载覆盖层"，**重启后仍使用 env var** 中的配置
- 用户需要同时更新 env var 才能让配置在重启后持久生效（面板里会有提示说明这一点）

#### 六、日志脱敏规则

修改 `database.py` 的 `set_global_setting()`：

```python
_SENSITIVE_KEYS = {"openai_compatible_api_key", "deepseek_api_key", "gemini_api_key"}

if key in _SENSITIVE_KEYS and value:
    masked = value[:4] + "****" + value[-4:] if len(value) > 8 else "****"
    log.info(f"已更新全局设置: {key} = {masked}")
else:
    log.info(f"已更新全局设置: {key} = {value}")
```
